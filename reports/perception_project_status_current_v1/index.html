PerceptionISP 목적·현재 진행상황·다음 단계

PerceptionISP가 왜 필요한지, 지금 어디까지 구현·검증됐는지, 다음 의사결정 전에 무엇을 완료해야 하는지를 하나의 보고서로 정리했습니다.

현재 결론: PerceptionISP는 sensor-aware software reference와 evidence pipeline으로서는 완성도가 높은 단계다. 다만 production ISP 또는 HumanISP 대비 일반 성능 우월성을 주장할 단계는 아니며, 다음 핵심은 native multi-exposure RAW·실센서 calibration·공정한 multi-seed task benchmark·temporal registration이다.

1. PerceptionISP의 목적

문제 정의

전통적인 ISP는 사람이 보기 자연스러운 RGB를 주 목표로 삼습니다. 하지만 detector·segmentation·SLAM 같은 machine perception은 예쁜 영상보다 경계 보존, 포화·노이즈·demosaic 불확실성, HDR source 선택, 시간 안정성 같은 정보를 필요로 합니다.

PerceptionISP는 같은 RAW에서 Human-view RGB와 machine-view RGB + Aux evidence를 동시에 만들어, 어느 정보가 downstream task에 도움이 되는지 공정하게 연구하는 software reference입니다.

성공의 정의

  • 하나의 sensor-domain source of truth와 감사 가능한 provenance
  • 동일 scene·RAW·annotation·모델·운영점의 matched comparison
  • 성능 gain과 FP·latency·calibration trade-off 동시 보고
  • 유리한 결과뿐 아니라 counterexample과 claim boundary 유지
  • native RAW와 pseudo-RAW를 절대 혼용하지 않음

범위

포함포함하지 않음
RAW normalization, HDR, CFA/demosaic, color, noise, edge/optics, temporal evidence, Human/Machine RGB, Aux tensor, detection/segmentation evaluation고수준 행동계획, 범용 detector 대체, 생산차량 ISP 품질 보장, native RAW 복원용 inverse ISP, 자동으로 보장되는 task 성능 향상
핵심 판단: “33개 map을 만들 수 있다”는 소프트웨어 실행 가능성이고, “33개 map이 실제 perception을 개선한다”는 별도의 held-out task 가설입니다.

2. 실제 코드 기준 아키텍처

핵심 설계는 “RGB를 더 예쁘게 만드는 ISP”가 아니라, RGB를 만들며 사라지는 sensor evidence와 불확실성을 공간 정렬된 map으로 보존하고 동일 task gate에서 효용을 검증하는 것입니다.

두 개의 truth를 분리합니다. native multi-exposure RAW는 exposure·gain·CFA·black/white·calibration의 source attestation이 있을 때 radiometric/HDR claim 후보가 됩니다. nuScenes RGB → CameraE2E 경로는 동일 RGB scene의 forward re-capture로 만든 pseudo-RAW이며, 파이프라인·metadata·temporal risk를 검증하지만 native RAW 복원 또는 실제 bracket 품질을 주장하지 않습니다.

왜 이 구조인가

  • 정보 보존: RGB formation에서 버려지는 exposure source, saturation, noise, CFA support, optics, timing을 side information으로 남깁니다.
  • 공간 정렬: 대부분의 Aux를 RGB와 같은 좌표계의 H×W map으로 만들어 early fusion과 pixel-level audit이 가능합니다.
  • 단조·해석 가능: risk가 커질수록 confidence가 낮아지는 bounded gate를 우선해 실패 방향을 추적할 수 있습니다.
  • 모듈성: calibration passthrough, physics-inspired model, project heuristic을 구분해 sensor profile과 task profile을 독립 교체할 수 있습니다.
  • 반증 가능성: 각 map은 적용 조건과 neutral fallback을 명시하고 group ablation·zero-Aux·matched RGB baseline으로 효용을 반증할 수 있습니다.
  • 제품 경계: camera-side low-level evidence까지만 담당하고 object semantics·BEV·planning은 downstream에 남깁니다.

3. 33개 Aux map: 실패 상황 → 수식 → 기대 효과

아래 33개 항목은 이름만 붙인 시각화가 아닙니다. public map contract, 실제 구현식, 선택 근거, 적용 조건을 같은 카드에 묶었으며 catalog와 이름 집합이 달라지면 테스트가 실패하도록 동기화했습니다.

선정 기준 1 · 관측 가능성RAW plane, metadata, calibration, 이전 state에서 계산 가능한 low-level evidence만 포함합니다. object class 같은 semantic label은 Aux가 아닙니다.
선정 기준 2 · 상보성단순 RGB가 숨기는 exposure·clipping·noise·CFA·optics·timing 실패축을 우선합니다. 서로 비슷한 map은 ablation 대상으로 남깁니다.
선정 기준 3 · 감사 가능성수식·값 의미·neutral fallback·소비 경로를 설명할 수 있어야 하며 최종 채택은 matched task evidence로 결정합니다.
근거 수준을 혼동하지 않습니다. Poisson–Gaussian noise, exposure normalization, structure tensor, calibration transfer는 물리·고전 추정 원리에서 출발합니다. 여러 항을 곱하거나 threshold로 묶은 confidence/flicker/likelihood는 이 프로젝트의 명시적 heuristic이며 fitted probability가 아닙니다.

6. 현재 한계와 위험

물리·데이터 한계

  • nuScenes는 processed auto-exposed JPEG이며 source exposure/gain/WB/CFA/black·white가 없습니다.
  • CameraE2E는 forward recapture simulator이고 inverse ISP나 JPEG clipping 복원기가 아닙니다.
  • 인접-frame HDR 근사는 registration·pose warp·optical flow·deghosting이 없는 stress stack입니다.
  • 기본 MTF/PSF/lens/noise map은 실센서 calibration을 대신하지 못합니다.
  • origin이 unknown인 metadata도 현재 기본 숫자로 계산될 수 있습니다. 따라서 값의 존재와 물리적 calibration validity를 분리하는 gate가 필요합니다.
  • 일부 native loader는 실제 mosaic/CFA를 읽어도 source exposure·gain을 완전히 복원하지 못하고 기본값을 사용합니다. “native CFA”와 “완전 calibrated RAW”는 같은 뜻이 아닙니다.
  • CameraE2E의 독립 호출 noise/FPN persistence는 실센서 temporal noise로 calibration되지 않았습니다.
  • HDR noise variance는 fusion weight별 분산을 엄밀히 전파하지 않고 fused signal에 단일 모델을 다시 적용하며, dark-current 항은 첫 exposure time만 사용합니다.
  • saturation과 clipping_distance는 fused recoverability가 아니라 any/max input-plane 기준이라 짧은 노출이 복구한 위치에도 보수적으로 반응할 수 있습니다.

성능·제품화 한계

  • HumanISP 대비 보편적 detector 우월성이 증명되지 않았습니다.
  • 33개 map 전체가 필요한지, 어떤 subset이 task별 최적인지 미확정입니다.
  • 현재 fast tensor는 전체 HDR·demosaic·color·formation 뒤에 만들어집니다. 따라서 실제 streaming early-output path가 아닙니다.
  • Fast-path latency는 모델 추정치이고 실제 hardware 측정치가 아닙니다.
  • 현재 320×180 showcase bundle은 전체 lossless trace 때문에 약 1.1GB로 큽니다.
  • float64 software reference를 embedded fixed-point ISP로 옮기는 설계는 아직 없습니다.
  • RuntimeController는 다음 설정을 제안하지만 closed-loop로 재적용하지 않습니다.
  • 감사에서 발견: extended tensor는 16채널인데 feature-distillation loader의 CHW/HWC 판별 집합에 16이 없어 16×H×W NPY를 오해석할 수 있습니다. 수정·회귀 테스트 전에는 해당 조합을 claim run에 사용하면 안 됩니다.
  • output_bit_depth, edge_confidence_floor, include_raw_like_tensor, color_shading_gain은 현재 core processing에서 실질적으로 소비되지 않습니다.
금지해야 할 표현: “nuScenes JPEG가 native RAW로 복원됐다”, “인접 JPEG가 실제 HDR bracket이다”, “ghost map이 deghosting을 완료했다”, “33개 Aux가 detector 성능을 자동 향상한다”, “현재 latency가 production hardware 성능이다”.
